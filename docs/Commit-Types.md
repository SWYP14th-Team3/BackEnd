# Git 커밋 타입 정리

## 1. 커밋 타입이란?

커밋 타입은 커밋 메시지 맨 앞에 작성하여 **어떤 종류의 변경인지 빠르게 알려주는 표시**입니다.

```text
타입: 변경 내용을 요약한 문장
```

예시는 다음과 같습니다.

```text
feat: 회원가입 기능 추가
fix: 회원 조회 실패 오류 수정
docs: API 실행 방법 작성
```

커밋 타입을 일관되게 사용하면 협업하는 사람이 커밋 제목만 보고도 변경 목적을 쉽게 이해할 수 있습니다.

## 2. 주요 커밋 타입

다음은 Conventional Commits에서 주로 사용하는 타입입니다.

| 타입 | 사용하는 상황 | 예시 |
|---|---|---|
| `feat` | 새로운 기능을 추가할 때 | `feat: 회원가입 기능 추가` |
| `fix` | 잘못 동작하는 기능이나 버그를 수정할 때 | `fix: 존재하지 않는 회원 조회 오류 수정` |
| `docs` | 문서만 추가하거나 수정할 때 | `docs: API 실행 방법 추가` |
| `style` | 기능 변화 없이 공백, 들여쓰기, 코드 형식 등을 변경할 때 | `style: 예외 처리 코드 들여쓰기 정리` |
| `refactor` | 기능은 그대로 유지하면서 코드 구조를 개선할 때 | `refactor: 회원 조회 로직을 서비스로 분리` |
| `test` | 테스트 코드를 추가하거나 수정할 때 | `test: 회원가입 서비스 단위 테스트 추가` |
| `build` | 빌드 도구, 라이브러리 또는 의존성을 변경할 때 | `build: MySQL 드라이버 의존성 추가` |
| `ci` | CI/CD 설정을 추가하거나 수정할 때 | `ci: GitHub Actions 테스트 워크플로 추가` |
| `perf` | 기능은 유지하면서 성능을 개선할 때 | `perf: 회원 목록 조회 쿼리 최적화` |
| `chore` | 위 타입에 포함되지 않는 프로젝트 관리 작업을 할 때 | `chore: Git 제외 파일 설정` |
| `revert` | 이전 커밋의 변경 사항을 되돌릴 때 | `revert: 회원가입 기능 변경 되돌리기` |

## 3. 타입별 상세 설명

### `feat`

사용자가 이용할 수 있는 새로운 기능이 생겼을 때 사용합니다.

```text
feat: 로그인 API 추가
feat: 게시글 작성 기능 추가
feat: 회원 프로필 조회 기능 추가
```

새로운 클래스나 파일이 생겼다고 무조건 `feat`인 것은 아닙니다. 실제 기능이 추가되었는지를 기준으로 판단합니다.

### `fix`

기존 기능이 잘못 동작하는 문제를 수정했을 때 사용합니다.

```text
fix: 중복 아이디가 저장되는 문제 수정
fix: 존재하지 않는 게시글 조회 시 500 오류가 발생하는 문제 수정
fix: 로그인 쿠키가 전달되지 않는 문제 수정
```

### `docs`

프로그램 동작에는 변화가 없고 문서만 변경했을 때 사용합니다.

```text
docs: 프로젝트 실행 방법 추가
docs: 회원 API 요청 예시 작성
docs: PR 템플릿 작성 가이드 보완
```

### `style`

코드 동작에는 영향을 주지 않고 코드 모양만 변경했을 때 사용합니다.

```text
style: 코드 들여쓰기 정리
style: 불필요한 공백 제거
style: import 순서 정리
```

CSS 디자인 변경을 의미하는 타입이 아니라는 점에 주의합니다.

### `refactor`

기존 기능과 결과는 그대로 유지하면서 내부 코드 구조를 개선했을 때 사용합니다.

```text
refactor: 공통 회원 검증 로직 분리
refactor: 생성자 주입 방식으로 의존성 주입 통일
refactor: 중복된 오류 응답 생성 코드 제거
```

기존 버그를 고쳤다면 `refactor`보다 `fix`가 더 적절합니다.

### `test`

테스트 코드만 추가하거나 수정했을 때 사용합니다.

```text
test: 회원가입 성공 테스트 추가
test: 중복 아이디 예외 테스트 추가
test: 애플리케이션 컨텍스트 테스트 수정
```

기능 코드와 테스트 코드를 함께 추가한 경우 커밋의 핵심 목적에 따라 `feat` 또는 `fix`를 사용할 수 있습니다.

### `build`

프로젝트 빌드 과정, 빌드 도구 또는 외부 의존성을 변경했을 때 사용합니다.

```text
build: Spring Validation 의존성 추가
build: Gradle Wrapper 버전 변경
build: 테스트 환경에 H2 데이터베이스 추가
```

대표적인 대상은 다음과 같습니다.

```text
build.gradle
settings.gradle
gradle-wrapper.properties
package.json
```

### `ci`

자동 빌드, 테스트 또는 배포 파이프라인 설정을 변경했을 때 사용합니다.

```text
ci: PR 생성 시 테스트가 실행되도록 설정
ci: main 브랜치 자동 배포 워크플로 추가
ci: Gradle 캐시 설정 추가
```

대표적인 대상은 GitHub Actions, Jenkins, CircleCI 같은 자동화 설정입니다.

### `perf`

기능의 결과는 유지하면서 실행 속도나 자원 사용량을 개선했을 때 사용합니다.

```text
perf: 회원 이메일 조회를 위한 인덱스 추가
perf: 게시글 목록 N+1 조회 문제 개선
perf: 반복적인 데이터베이스 조회 제거
```

### `chore`

기능, 버그 수정, 문서, 테스트 등에 정확히 해당하지 않는 프로젝트 관리 작업에 사용합니다.

```text
chore: .gitignore 설정 추가
chore: 로컬 개발용 Docker Compose 구성
chore: 사용하지 않는 설정 파일 제거
```

무엇이든 `chore`로 처리하기보다는 다른 타입에 정확히 해당하는지 먼저 확인하는 것이 좋습니다.

### `revert`

이전에 적용한 커밋을 취소할 때 사용합니다.

```text
revert: 회원가입 응답 형식 변경 되돌리기
```

어떤 커밋을 되돌렸는지 커밋 본문에 커밋 해시나 이유를 함께 기록하면 좋습니다.

## 4. 팀에서 추가로 사용할 수 있는 타입

다음 타입은 대표적인 Conventional Commits 타입이라기보다 팀 규칙에 따라 선택적으로 사용할 수 있습니다.

| 타입 | 사용하는 상황 | 예시 |
|---|---|---|
| `config` | 애플리케이션 실행 설정 변경 | `config: 개발 환경 데이터베이스 설정 추가` |
| `db` | DB 스키마, 인덱스 또는 초기 데이터 변경 | `db: 회원 이메일 인덱스 추가` |
| `security` | 인증, 인가 또는 보안 정책 변경 | `security: 허용된 출처로 CORS 정책 제한` |
| `remove` | 사용하지 않는 코드나 파일 제거 | `remove: 사용하지 않는 회원 DTO 삭제` |
| `rename` | 클래스, 변수, 파일 또는 경로 이름 변경 | `rename: MemberDto를 MemberResponse로 변경` |
| `hotfix` | 운영 환경의 긴급한 문제 수정 | `hotfix: 로그인 불가 오류 긴급 수정` |

팀 자체 타입을 너무 많이 만들면 구성원마다 판단 기준이 달라질 수 있습니다. 특별한 이유가 없다면 주요 타입을 먼저 사용하는 것을 권장합니다.

예를 들어 다음처럼 기존 타입으로도 충분히 표현할 수 있습니다.

```text
config 대신 chore 또는 feat
db 대신 feat, fix 또는 perf
security 대신 feat 또는 fix
remove와 rename 대신 refactor
hotfix 대신 fix
```

## 5. 타입 선택 방법

타입 선택이 어려울 때는 다음 기준을 사용합니다.

```text
사용자가 사용할 새로운 기능이 생겼는가?        → feat
잘못 동작하던 기능을 고쳤는가?                 → fix
동작은 같고 내부 코드 구조만 개선했는가?        → refactor
성능만 개선했는가?                             → perf
문서만 변경했는가?                             → docs
테스트만 변경했는가?                           → test
코드 모양만 정리했는가?                        → style
라이브러리나 Gradle 설정을 변경했는가?          → build
자동 빌드 또는 배포 설정을 변경했는가?          → ci
이전 커밋을 되돌렸는가?                         → revert
어디에도 정확히 포함되지 않는 관리 작업인가?    → chore
```

## 6. 커밋 메시지 작성 형식

기본 형식은 다음과 같습니다.

```text
타입: 무엇을 변경했는지 작성
```

좋은 예시는 다음과 같습니다.

```text
feat: 회원가입 API 추가
fix: 중복 아이디 검증 오류 수정
refactor: 공통 예외 처리 로직 분리
test: 회원 조회 실패 테스트 추가
```

피하면 좋은 예시는 다음과 같습니다.

```text
수정
작업 완료
버그 고침
코드 변경
```

변경 내용을 구체적으로 표현하지 않아서 다른 사람이 커밋의 목적을 파악하기 어렵기 때문입니다.

## 7. 선택적으로 Scope 사용하기

변경된 영역을 더 명확히 보여주고 싶다면 타입 뒤에 Scope를 추가할 수 있습니다.

```text
타입(영역): 변경 내용
```

예시는 다음과 같습니다.

```text
feat(member): 회원가입 API 추가
fix(auth): 만료된 토큰 검증 오류 수정
refactor(exception): 전역 예외 처리 구조 개선
test(post): 게시글 삭제 서비스 테스트 추가
```

Scope는 반드시 사용할 필요는 없습니다. 사용한다면 팀에서 영역 이름을 통일하는 것이 좋습니다.

## 8. 호환성이 깨지는 변경 표시하기

기존 API나 사용 방법이 더 이상 호환되지 않는 변경은 타입 뒤에 `!`를 붙여 표시할 수 있습니다.

```text
feat!: 로그인 응답 형식 변경
```

Scope와 함께 사용할 수도 있습니다.

```text
feat(auth)!: 로그인 토큰 전달 방식을 쿠키로 변경
```

이 경우 커밋 본문에 무엇이 달라졌고 사용자가 어떻게 대응해야 하는지 반드시 설명하는 것이 좋습니다.

## 9. 이 프로젝트의 권장 타입

이 프로젝트에서는 우선 다음 타입을 일관되게 사용하는 것을 권장합니다.

```text
feat      새로운 기능
fix       버그 수정
refactor  코드 구조 개선
docs      문서 변경
test      테스트 변경
build     빌드 및 의존성 변경
ci        CI/CD 변경
chore     기타 프로젝트 관리
style     코드 형식 변경
perf      성능 개선
revert    이전 커밋 되돌리기
```

가장 중요한 것은 타입의 개수보다 **팀 구성원이 동일한 기준으로 꾸준히 사용하는 것**입니다.
